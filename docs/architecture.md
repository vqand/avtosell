# 5VITO — Архитектура и документация проекта

## 1. Архитектура системы

```
┌────────────────────┐        HTTPS / JSON        ┌────────────────────┐
│   React SPA (Vite)  │  ───────────────────────▶  │   PHP REST API     │
│  - React Router     │   Authorization: Bearer    │  - Фронт-контроллер │
│  - Контекст auth    │  ◀───────────────────────  │  - Router + MW      │
│  - Клиент fetch     │                            │  - Контроллеры      │
└────────────────────┘                            │  - Модели (PDO)     │
                                                   └─────────┬──────────┘
                                                             │ подготовленные
                                                             │ выражения
                                                       ┌─────▼──────┐
                                                       │   MySQL     │
                                                       └────────────┘
```

- **Stateless API**: серверных сессий нет; JWT несёт `sub`, `role`, `name`.
- **Единый источник истины**: база данных. SPA хранит только состояние UI и
  токен (в `localStorage`).
- **Разделение ответственности**: тонкие контроллеры, доступ к данным изолирован
  в моделях, сквозные задачи (авторизация, CORS, ошибки) — в middleware/бутстрапе.

### Слои бэкенда (MVC)

| Слой        | Ответственность                                  | Расположение            |
|-------------|--------------------------------------------------|-------------------------|
| Маршрутизация | Сопоставление метод+путь с обработчиком, запуск middleware | `src/Core/Router.php` |
| Middleware  | Аутентификация, контроль ролей (RBAC)            | `src/Middleware/`       |
| Контроллеры | Валидация входных данных, оркестрация, формирование ответа | `src/Controllers/` |
| Модели      | Весь SQL (подготовленные выражения)              | `src/Models/`           |
| Core-сервисы| DB, JWT, Request, Response, Validator            | `src/Core/`             |

### Обоснование выбора технологий

- **React + Vite** — компонентная модель идеально подходит для экранов каталога,
  карточки и админки; Vite даёт мгновенный HMR и компактную сборку.
- **PHP (без фреймворка)** — ТЗ требует PHP + MVC; самописный микро-MVC делает
  проект прозрачным для академической проверки, демонстрируя при этом
  маршрутизацию, middleware, валидацию и PDO. Работает на стандартном XAMPP без
  единой зависимости Composer.
- **MySQL** — реляционные данные (пользователи, авто, бренды, избранное)
  естественно ложатся на нормализованную схему; полнотекстовый индекс обслуживает
  строку поиска.
- **JWT** — stateless-авторизация хорошо подходит для связки SPA + REST API и
  масштабируется горизонтально без общего хранилища сессий.

## 2. Поток аутентификации и авторизации

1. `POST /auth/login` проверяет учётные данные (`password_verify`) и возвращает
   подписанный JWT (HS256, срок жизни `JWT_TTL` секунд).
2. SPA сохраняет токен и отправляет его в заголовке `Authorization: Bearer …`.
3. `AuthMiddleware` проверяет подпись и срок действия, заполняет `request.auth`.
4. `AdminMiddleware` дополнительно требует `role === 'admin'`.
5. Принадлежность ресурса (редактирование/удаление объявления) проверяется в
   контроллере.

## 3. Стратегия безопасности

| Угроза                       | Мера защиты                                                          |
|------------------------------|---------------------------------------------------------------------|
| Кража паролей                | bcrypt через `password_hash()` / `password_verify()`                |
| SQL-инъекции                 | Подготовленные выражения PDO везде; `ATTR_EMULATE_PREPARES=false`    |
| XSS                          | React автоматически экранирует; API отдаёт только JSON, никогда HTML |
| Сломанная аутентификация     | Подписанный JWT, `hash_equals`, контроль срока действия              |
| Сломанный контроль доступа   | Middleware auth + admin; проверка владельца на уровне ресурса        |
| Вредоносные загрузки         | Определение MIME (`finfo`), лимит размера, случайные имена, без exec |
| CSRF                         | Схема с Bearer-токеном (не cookie) не подвержена CSRF; CORS ограничен|
| Утечка чувствительных данных | `password_hash` не сериализуется; email продавца скрыт в публичной карточке |
| Ошибки конфигурации          | Секреты в `.env`; `APP_DEBUG=false` скрывает трассировки в проде     |

## 4. Стратегия тестирования

- **Модульное (Unit)** — `Validator`, `JWT::encode/decode`, построители запросов
  моделей (SQL фильтрации).
- **Интеграционное / API** — прогон каждого эндпоинта через PHPUnit с тестовой
  базой либо коллекцию Postman/Newman (успешный путь + случаи 401/403/422).
- **UI** — React Testing Library для компонентов (CarCard, фильтры, формы);
  Cypress/Playwright для сквозных сценариев (вход → поиск → избранное → создание).
- **Безопасность** — проверка RBAC (пользователь не может обращаться к `/admin/*`),
  проверка владельца, попытки SQL-инъекций в параметрах фильтра, отказ при
  неверном типе/размере загрузки.
- **Производительность** — `EXPLAIN` для запроса каталога; подтверждение
  использования индексов.

> Реализованная проверка: набор сквозных API-тестов покрывает регистрацию
> (включая дубликат → 409 и валидацию → 422), вход (пароль/телефон/неверный → 401),
> CRUD объявлений, модерацию, RBAC (→ 403) и избранное.

## 5. План развёртывания

**Разработка**: `php -S` для API + dev-сервер Vite (проксирует `/api`).

**Продакшен (один хост / в стиле XAMPP)**
1. Виртуальный хост Apache с `DocumentRoot → backend/public` и `AllowOverride All`.
2. `npm run build`; раздавать `frontend/dist` как статику (тот же хост или CDN),
   задать `VITE_API_BASE` равным адресу API.
3. MySQL с отдельным пользователем с минимальными правами (не `root`).
4. `.env`: надёжный `JWT_SECRET`, `APP_DEBUG=false`, реальный `CORS_ORIGIN`.
5. HTTPS через обратный прокси (Nginx/Caddy) или Apache + Let's Encrypt.

**Облачный вариант**: контейнеризация (образ PHP-FPM + Nginx, статика фронтенда
на CDN, управляемая MySQL вроде RDS/PlanetScale). Горизонтальное масштабирование
тривиально, поскольку API stateless.

## 6. Сопровождение и мониторинг

- **Резервные копии**: ночной `mysqldump` (или автоматические снапшоты
  управляемой БД); проверять восстановление.
- **Мониторинг**: эндпоинт `/api/health` для проверки доступности; агрегация
  логов ошибок PHP; отслеживание доли ответов 5xx и медленных запросов (slow-query
  log в MySQL).
- **Аудит**: таблица `admin_logs` фиксирует каждое действие администратора.

## 7. Дорожная карта (дальнейшее развитие)

Реализовано как MVP, намеренно оставлено для расширения:

- Поток подтверждения по SMS (экраны макета 1b–1c) — сейчас вход по паролю.
- Сохранение заявок на покупку (сейчас подтверждение на стороне клиента).
- Refresh-токены / список отзыва токенов.
- Элементы управления пагинацией в UI каталога (API уже отдаёт страницы).
- Управление галереей из нескольких фото с drag-and-drop.
- Сохранённые поиски и email-уведомления.
- Ограничение частоты запросов (rate limiting) и CAPTCHA на эндпоинтах входа.
- Полный набор автотестов + CI-конвейер.
