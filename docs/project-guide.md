# 5VITO — Полное руководство по проекту

Документ объединяет все части проекта: разбор реализации фронтенда и бэкенда,
жизненный цикл запроса, пользовательские сценарии и структуру финальной
презентации. Для архитектуры см. [architecture.md](architecture.md), для БД —
[database-er.md](database-er.md), для API — [api-spec.md](api-spec.md).

---

## 1. Общее описание

**5VITO** — учебный/портфолио-проект полнофункционального маркетплейса по продаже
автомобилей в стиле Avito. Пользователи регистрируются, размещают объявления о
продаже авто, ищут и фильтруют каталог, добавляют объявления в избранное и
оформляют заявку на покупку. Администраторы модерируют объявления, управляют
пользователями, брендами и просматривают жалобы.

Технологический стек: **React (Vite)** на фронтенде, **PHP (собственный MVC)** на
бэкенде, **MySQL** в качестве базы данных, **JWT** для авторизации.

## 2. Жизненный цикл запроса (бэкенд)

Все запросы к API проходят через единый фронт-контроллер `backend/public/index.php`:

1. **Бутстрап** — `App::registerAutoloader()` подключает PSR-4-подобный
   автозагрузчик; `config/config.php` читает `.env`.
2. **CORS** — `App::handleCors()` выставляет заголовки и сразу отвечает на
   `OPTIONS` (preflight).
3. **Подключение к БД** — `Database::init()` создаёт единственное соединение PDO
   (режим исключений, эмуляция подготовленных выражений отключена).
4. **Объект запроса** — `new Request()` разбирает метод, путь, query-параметры,
   JSON-тело и Bearer-токен.
5. **Маршрутизация** — `Router::dispatch()` находит маршрут по методу и шаблону
   (`/cars/{id}` → регулярное выражение с именованными группами).
6. **Middleware** — для защищённых маршрутов выполняются `AuthMiddleware`
   (проверка JWT) и `AdminMiddleware` (проверка роли). Любой из них может
   прервать выполнение, вернув ошибку.
7. **Контроллер** — валидирует данные (`Validator`), обращается к моделям и
   формирует ответ через `Response::json()`.
8. **Модель** — выполняет SQL исключительно через подготовленные выражения.

Ошибки перехватываются глобальным обработчиком (`App::registerErrorHandler`):
в режиме отладки возвращается сообщение и трассировка, в проде — нейтральное
«Internal server error».

### Ключевые файлы бэкенда

| Файл                         | Роль                                              |
|------------------------------|---------------------------------------------------|
| `public/index.php`           | Фронт-контроллер: бутстрап + объявление маршрутов |
| `public/router.php`          | Скрипт-роутер для встроенного сервера PHP          |
| `src/Core/Router.php`        | Маршрутизатор с поддержкой `{param}` и middleware |
| `src/Core/Database.php`      | Синглтон PDO                                       |
| `src/Core/Request.php`       | Разбор входящего запроса                           |
| `src/Core/Response.php`      | Хелпер JSON-ответов                                |
| `src/Core/JWT.php`           | Кодирование/декодирование HS256 без зависимостей  |
| `src/Core/Validator.php`     | Валидатор на основе правил                         |
| `src/Middleware/*`           | Проверки авторизации и прав администратора         |
| `src/Controllers/*`          | Логика обработки: Auth, Car, Favorite, Brand, Upload, Admin |
| `src/Models/*`               | Доступ к данным: User, Car, Brand, Favorite, Report, Stats |

## 3. Архитектура фронтенда

SPA на React с маршрутизацией через `react-router-dom`. Точка входа —
`src/main.jsx`, которая оборачивает приложение в `BrowserRouter` и `AuthProvider`.

### Маршруты (`src/App.jsx`)

| Путь                     | Страница            | Доступ            |
|--------------------------|---------------------|-------------------|
| `/login`, `/register`    | Вход / регистрация  | публичный (без шапки) |
| `/`                      | Каталог             | публичный         |
| `/cars/:id`              | Карточка авто       | публичный         |
| `/cars/:id/buy`          | Заявка на покупку   | публичный         |
| `/success`               | Успех               | публичный         |
| `/favorites`             | Избранное           | требуется вход    |
| `/profile`               | Профиль             | требуется вход    |
| `/my-listings`           | Мои объявления      | требуется вход    |
| `/listings/new`          | Новое объявление    | требуется вход    |
| `/listings/:id/edit`     | Редактирование      | требуется вход    |
| `/admin/*`               | Админ-панель        | только админ      |

### Управление состоянием

- **`AuthContext`** (`src/context/AuthContext.jsx`) — хранит текущего пользователя
  и токен, предоставляет `login`, `register`, `logout`, восстанавливает сессию по
  токену из `localStorage` при загрузке.
- **`ProtectedRoute`** — оборачивает приватные маршруты, перенаправляет
  неавторизованных на `/login`, а не-админов с `/admin` — на главную.
- Локальное состояние страниц — через хуки `useState`/`useEffect` (фильтры
  каталога, формы).

### Клиент API (`src/api/client.js`)

Тонкая обёртка над `fetch`: автоматически добавляет заголовок
`Authorization: Bearer`, сериализует JSON, бросает `ApiError` с HTTP-статусом и
полями ошибок валидации. Хелпер `qs()` собирает строку запроса из объекта
фильтров. Метод `upload()` отправляет `FormData` для загрузки изображений.

### Соответствие макетам Figma

| Экран макета                         | Реализация              |
|--------------------------------------|-------------------------|
| Войти (телефон или почта + пароль)   | `pages/Login.jsx`       |
| Каталог с фильтрами и поиском        | `pages/Catalog.jsx`     |
| Карточка авто с полной спецификацией | `pages/CarDetail.jsx`   |
| Заявка на осмотр/покупку             | `pages/PurchaseRequest.jsx` |
| «Успешно!»                           | `pages/Success.jsx`     |

Стилизация — единый файл `src/index.css` в минималистичной чёрно-белой палитре,
повторяющей макет.

## 4. Пользовательские сценарии

### Покупатель
1. Открывает каталог, ищет по строке поиска или фильтрам (бренд, цена, привод,
   страна бренда).
2. Открывает карточку авто, изучает характеристики и фото.
3. Добавляет понравившееся в избранное (требуется вход).
4. Оформляет «Заявку на покупку» — телефон и дату осмотра → экран «Успешно!».

### Продавец
1. Регистрируется и входит.
2. В «Мои объявления» нажимает «Разместить авто», заполняет форму и загружает фото.
3. Объявление уходит на модерацию (статус `pending`) и не видно в публичном
   каталоге до одобрения.
4. Может редактировать и удалять свои объявления.

### Администратор
1. Входит под админ-аккаунтом и попадает в `/admin`.
2. На дашборде — статистика (пользователи, авто, на модерации, жалобы).
3. Модерирует объявления (одобрить/отклонить/удалить).
4. Управляет пользователями (роль, блокировка, удаление), брендами и жалобами.

## 5. Безопасность (кратко)

- Пароли — `password_hash()` (bcrypt), проверка — `password_verify()`.
- Все SQL — подготовленные выражения PDO (эмуляция отключена).
- JWT подписан HS256, проверка подписи через `hash_equals`, контроль `exp`.
- RBAC: `AdminMiddleware` на каждом маршруте `/api/admin/*` + проверка владельца
  ресурса в контроллерах.
- Загрузка файлов: проверка MIME через `finfo`, лимит размера, случайные имена.
- React экранирует вывод (XSS); API отдаёт только JSON.

Подробнее — раздел «Стратегия безопасности» в [architecture.md](architecture.md).

## 6. Тестирование

Реализованный сквозной набор API-тестов подтверждает:
регистрацию (включая дубликат → 409 и ошибку валидации → 422), вход
(пароль / телефон / неверный → 401), создание объявления (статус `pending`),
сокрытие неодобренного из каталога, одобрение администратором и появление в
каталоге, RBAC (пользователь в админке → 403), избранное (добавление/список/
удаление), обновление профиля, редактирование, запрет удаления чужого объявления
(→ 403), удаление владельцем (→ затем 404), а также загрузку изображения
(201, файл сохранён, привязан к авто, отдаётся как статика).

План дальнейшего тестирования (unit / интеграционное / UI / безопасность /
производительность) — раздел «Стратегия тестирования» в
[architecture.md](architecture.md).

4. **База данных** — ER-диаграмма, нормализация, индексы.
